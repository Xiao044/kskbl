## 2. 技术架构总览

从技术实现角度看，本项目采用的是“前端可视化大屏 + FastAPI 后端服务 + 抓包解析引擎 + WebSocket 实时推送 + 大模型智能分析”的复合型技术架构。

可以将技术架构概括为：

`Vue 3 前端应用 + ECharts 可视化 + FastAPI 服务层 + Python 抓包解析引擎 + 内存状态中心 + WebSocket 实时通信 + DeepSeek 大模型能力`

### 2.1 前端技术栈

前端采用以下技术组合：

- Vue 3
  - 作为前端主框架
  - 用于构建组件化页面结构

- Vue CLI 5
  - 用于项目构建与开发服务

- ECharts 6
  - 用于折线图、柱状图、饼图、雷达图等数据可视化展示

- 原生 WebSocket
  - 用于实时接收流量与告警数据
  - 用于 AI 问答双向通信

- Clay 风格自定义 CSS 设计系统
  - 用于构建整体视觉风格
  - 通过统一变量体系管理颜色、毛玻璃、阴影与动效

### 2.2 后端技术栈

后端采用以下技术组合：

- Python
  - 作为主要开发语言

- FastAPI
  - 作为 Web 服务框架
  - 同时支持 REST API 与 WebSocket

- AsyncIO
  - 用于异步队列消费和实时通信协作

- threading
  - 用于抓包解析后台线程

- dpkt
  - 用于离线 PCAP / PCAPNG 报文解析

- OpenAI Python SDK 兼容调用方式
  - 用于接入 DeepSeek 大模型接口

- 内存态状态管理
  - 当前未引入数据库，采用 Python 内存结构维护运行时状态

从技术实现分层角度，可以进一步拆成六层：

1. 表现层
   - Vue 组件、图表、地图、聊天面板

2. 交互层
   - 路由切换、抽屉交互、登录控制、会话控制

3. 通信层
   - HTTP API
   - WebSocket 双通道

4. 服务层
   - FastAPI 路由
   - AI 会话服务
   - 状态查询服务

5. 状态层
   - 全局流量状态
   - 告警缓冲
   - 历史缓存
   - AI 会话上下文

6. 数据处理层
   - 抓包解析
   - 流量聚合
   - 协议识别
   - 威胁规则识别

## 5. 分层架构设计

## 5.1 数据源层

数据源层当前以离线抓包文件为主要输入。

输入文件：

- `sample.pcap`

这一层的职责是：

- 提供原始网络数据

- 为后端构造可重复演示、可稳定回放的数据基础

这一设计适合答辩和演示，因为：

- 可控性强
- 数据可重复
- 不依赖现场真实网络环境

---

## 5.2 数据解析与处理层

该层由 `engine.py` 负责，是整个系统的数据处理核心。

### 4.2.1 核心职责

`engine.py` 的职责主要包括：

- 从 PCAP 文件中逐包读取网络数据
- 识别 IPv4 / IPv6 数据包
- 提取源 IP、目的 IP、流量长度、端口信息
- 识别协议类型
- 按时间窗口聚合当前活跃流
- 检测异常并生成威胁告警
- 构建当前窗口数据与 Top-K 数据
- 将结果放入缓冲队列供后端消费

### 4.2.2 协议识别机制

协议识别采用双层机制：

1. 基于端口的 L4 映射识别
2. 基于载荷特征的简化 DPI 识别

例如：

- 80 -> HTTP
- 443 -> HTTPS
- 22 -> SSH
- 445 -> SMB
- 负载以 `GET` / `POST` / `HTTP/` 开头时识别为 HTTP
- 以 TLS 握手字节开头时识别为 HTTPS/TLS

这样设计的意义在于：

- 兼顾识别速度与可解释性
- 即使端口不标准，也能在一定程度上识别业务协议

### 4.2.3 流量窗口聚合

系统采用近实时滑动窗口思路，以 1 秒为主要时间粒度进行流量切片。

每个时间窗口内，`engine.py` 会生成：

- `current_data`
  - 当前窗口内的热点流量数据
- `topk_data`
  - 全局累积视角下的 Top-K 大流量节点
- `window_alerts`
  - 当前窗口识别出的告警

这种机制的优点：

- 能模拟实时场景
- 便于大屏秒级刷新
- 后端与前端都可以使用统一时间粒度的数据

### 4.2.4 告警检测策略

当前告警检测基于轻量规则引擎实现，按窗口内的流量特征进行判断。

主要类型包括：

- 容量耗尽型 DDoS
- 大范围端口探测
- 疑似勒索蠕虫传播
- 高频机器爬虫
- 未知加密隧道外发

输出字段包括：

- 时间
- 类型
- 源 IP / 目的 IP
- 所属区域
- 地理映射信息
- 字节数
- 包数
- 风险等级
- 威胁类别

## 5.3 后端服务与状态管理层

这一层由 `main.py`、`core.py`、`routers.py` 共同组成。

它是整个系统的“中枢层”，负责：

- 生命周期管理
- 全局状态维护
- API / WebSocket 暴露
- AI 所需的数据组织与工具调用

### 4.3.1 服务入口：main.py

`main.py` 是后端应用入口。

其核心作用：

- 创建 FastAPI 实例
- 配置 CORS
- 使用 `lifespan` 管理后台任务
- 在启动时挂载：
  - 抓包解析线程
  - 队列消费异步任务

架构意义：

- 将数据解析与 Web 服务解耦
- 后端既负责对外响应，也负责内部异步数据刷新

### 4.3.2 状态中心：core.py

`core.py` 是后端架构的基础设施层。

它维护了整个系统运行所依赖的共享状态与基础能力。

#### 4.3.2.1 全局共享状态

当前全局状态包含：

- `current_flow`
  - 当前窗口内的活跃流量
- `topk_flow`
  - 当前累计 Top-K 流量
- `current_alerts`
  - 当前待推送的告警缓冲
- `recent_alerts`
  - 近期告警历史缓冲
- `flow_history`
  - 近期流量历史缓冲

这些状态由解析引擎持续更新，由接口层统一读取。

#### 4.3.2.2 并发控制

由于后端同时存在：

- 抓包线程
- 队列消费协程
- HTTP 请求
- WebSocket 请求

所以系统使用线程锁 `lock` 对共享状态访问进行保护，避免读写冲突。

#### 4.3.2.3 缓冲机制

系统引入：

- `playback_queue`
  - 用于解耦解析生产者与状态消费者
- LRU 流缓存
  - 控制全局流聚合规模
- 最近告警与流量历史缓冲
  - 为 AI 和历史查询提供基础

这种设计保证了系统既能实时运行，又能在一定程度上保留上下文。

### 4.3.3 接口层：routers.py

`routers.py` 是后端的业务出口层。

它面向前端和 AI 模块提供统一接口能力。

主要路由包括：

- `/api/topk`
- `/api/zone-stats`
- `/api/system-stats`
- `/api/threat-geo`
- `/ws/flow`
- `/ws/chat`

这一层的职责不是做底层解析，而是从状态中心取数、组织结果、对外输出。

## 6. 技术运行架构

这一部分强调系统在运行时是如何依赖不同技术模块协同工作的。

### 6.1 后端运行架构

后端是一个“同步线程 + 异步服务”混合架构。

内部由两类执行单元组成：

1. 抓包解析线程
   - 负责持续读取 PCAP 数据
   - 进行报文解析和窗口聚合

2. FastAPI 主服务 + AsyncIO 协程
   - 负责 HTTP API
   - 负责 WebSocket 通信
   - 负责队列消费

这种架构的优点是：

- 把 CPU 密集型解析过程与 Web 服务响应过程分离
- 避免阻塞主服务线程
- 更适合持续实时刷新场景

### 6.2 前端运行架构

前端本质是一个单页应用架构。

其运行方式如下：

1. `App.vue` 作为主容器加载
2. 登录成功后建立实时 WebSocket
3. 主容器统一接收流量与告警数据
4. 根据当前页面切换不同业务组件
5. 各组件根据 props 或接口数据完成独立渲染

同时，前端采用异步组件加载：

- `Dashboard`
- `Realtime`
- `Rank`
- `History`
- `Alert`
- `Chat`

这样做降低了首屏加载体积，提升了交互流畅度。

### 6.3 通信架构

系统通信采用“HTTP + WebSocket”双轨并行结构。

#### HTTP 负责：

- Top-K 查询
- 区域统计
- 系统资源状态
- 校园地图地理点数据

#### WebSocket 负责：

- 实时流量与告警推送
- AI 问答交互

这种分工方式比单纯全部用 HTTP 更适合实时看板，也比全部用 WebSocket 更便于调试与维护。

### 6.4 AI 技术运行架构

AI 模块在技术上采用“会话态 + 工具调用 + 大模型推理”的组合架构。

完整流程如下：

1. 前端 `Chat.vue` 发送用户问题
2. 后端 `/ws/chat` 接收请求
3. 后端从状态中心抽取系统快照
4. 将快照组织成结构化 system prompt
5. 挂载工具定义给模型
6. 若模型需要更多数据，则调用后端工具函数
7. 工具函数从全局状态中读取信息
8. 把工具结果再喂给模型
9. 返回最终答案给前端

这意味着 AI 技术架构不是简单 Prompt，而是具备检索与推理闭环的轻量级 Agent 架构。

### 6.5 性能优化架构

为了避免大屏页面卡顿，当前项目在技术层面已经引入多项优化策略：

- 前端页面组件异步加载
- ECharts 按需异步导入
- WebSocket 数据帧合并刷新
- 动画期间降低毛玻璃与阴影渲染成本
- 聊天面板与侧栏动画分离
- 后端状态缓冲上限控制
- LRU 缓存防止流量状态无限膨胀

这些优化共同构成了项目的性能保障架构。

## 7. 后端内部模块协作关系

从模块协作角度，后端内部关系如下：

### 5.1 engine.py -> core.py

`engine.py` 不直接服务前端，而是将解析结果投递到队列中，再由消费者写入 `core.py` 中的全局状态。

作用：

- 解耦实时数据生产与外部接口调用
- 降低直接写状态的时序复杂度

### 5.2 core.py -> routers.py

`routers.py` 从 `core.py` 中读取：

- 当前流量
- 当前告警
- Top-K
- 最近告警
- 历史流量

然后对外组织为：

- HTTP JSON 响应
- WebSocket 实时推送
- AI 工具调用返回结果

### 5.3 main.py -> engine.py / routers.py

`main.py` 负责把这两部分拼装起来：

- 启动 `engine.py` 的后台执行
- 挂载 `routers.py` 的接口服务

因此 `main.py` 是装配层，而不是业务层。

## 10. 前后端通信架构

系统前后端通信采用“双通道架构”：

1. HTTP API
2. WebSocket 实时推送

## 7.1 HTTP 通道

适用于：

- 查询型请求
- 非高频更新数据
- 页面主动拉取数据

例如：

- `/api/topk`
- `/api/zone-stats`
- `/api/system-stats`
- `/api/threat-geo`

特点：

- 语义清晰
- 易于调试
- 适合按需获取

## 7.2 WebSocket 通道

适用于：

- 高频实时流量推送
- 实时告警广播
- AI 对话交互

包括：

- `/ws/flow`
  - 面向前端实时业务数据更新
- `/ws/chat`
  - 面向 AI 问答交互

特点：

- 延迟低
- 实时性强
- 适合连续状态同步

这种双通道设计使系统能同时兼顾：

- 实时大屏刷新
- 查询型业务接口
- 多轮智能对话

## 11. AI 架构设计

AI 模块是本项目在传统态势看板之上增加的智能层。

## 8.1 接入方式

AI 模块通过 `/ws/chat` 与前端交互，并在后端使用大模型客户端进行调用。

## 8.2 会话架构

每个 WebSocket 连接都维护独立会话：

- 独立历史记录
- 最近多轮上下文
- 独立重置能力

这意味着系统不是全局广播式聊天，而是“按连接隔离”的多轮问答结构。

## 8.3 上下文构建

后端每轮会构建结构化系统快照，包括：

- 高危告警数
- 中危告警数
- 最近告警数
- 最近高危事件
- 当前 top talker
- 当前异常区域

这样 AI 回答能直接与系统当前状态绑定。

## 8.4 工具调用架构

AI 可通过工具函数主动查询系统数据。

当前工具包括：

- `get_topk`
- `get_zone_stats`
- `get_system_stats`
- `get_latest_alerts`
- `get_ip_history`

所以 AI 架构本质上是：

`对话接口 + 系统快照 + 工具调用 + 多轮历史`

而不是简单单轮聊天机器人。

## 12. 运行时数据流

系统运行时的数据流可以分为三条主链路：

## 9.1 实时业务数据链路

1. `sample.pcap`
2. `engine.py` 解析
3. 写入 `playback_queue`
4. `queue_consumer()` 消费
5. 更新 `global_state`
6. `/ws/flow` 每秒推送给前端
7. `App.vue` 接收后分发到页面

## 9.2 查询型接口链路

1. 前端页面或组件发起 HTTP 请求
2. `routers.py` 从 `global_state` 读取对应数据
3. 返回 JSON 响应
4. 页面根据结果渲染图表或表格

## 9.3 AI 问答链路

1. 用户在 `Chat.vue` 中输入问题
2. 前端通过 `/ws/chat` 发送到后端
3. 后端构建结构化快照
4. 大模型根据问题决定是否调用工具
5. 工具从 `global_state` 读取系统数据
6. 模型生成最终回答
7. 回答回传到前端聊天界面

## 13. 部署与运行架构

当前项目是单体式前后端分离部署结构。

### 10.1 后端运行方式

后端基于 FastAPI + Uvicorn 运行，监听：

- `0.0.0.0:8000`

支持局域网内设备访问。

- 独立数据库层
- 独立缓存层
- 独立模型服务层
- 独立日志与监控层

## 14. 技术架构总结

从技术架构角度，本项目的技术实现具备以下特点：

- 前端使用 Vue 3 + ECharts 构建可视化大屏
- 后端使用 FastAPI + AsyncIO + threading 构建实时服务
- 数据处理使用 dpkt 完成网络报文解析
- 系统运行依赖统一的内存状态中心
- 实时数据使用 WebSocket 推送
- 查询数据使用 HTTP 接口输出
- AI 采用“大模型 + 工具调用 + 多轮会话”的增强型架构

因此，本项目不是单纯的可视化页面，也不是单纯的抓包分析程序，而是一套具备完整技术链条的校园网络态势感知系统。

## 15. 架构特征总结

从系统架构角度看，本项目具有以下特征：

### 11.1 前后端分离

前端负责可视化与交互，后端负责解析、状态维护与服务输出，边界清晰。

### 11.2 数据驱动

整个系统围绕数据状态中心运行，页面与 AI 都从统一状态中心取数，保证一致性。

### 11.3 实时与查询并存

通过 WebSocket 实现实时更新，通过 HTTP 提供查询接口，兼顾动态性与可控性。

### 11.4 可扩展的智能分析层

AI 模块已经具备上下文和工具能力，为后续进一步扩展到智能运维、智能告警解释、自动处置建议提供了架构基础。

### 11.5 展示型与工程型兼顾

架构既能支撑工程逻辑，也非常适合可视化展示和答辩演示。

## 16. 一句话架构总结

本项目构建了一套以抓包解析引擎为数据核心、以全局状态中心为中枢、以前端多页面大屏为展示界面、以大模型问答为智能增强层的智慧校园网络态势感知系统架构。
